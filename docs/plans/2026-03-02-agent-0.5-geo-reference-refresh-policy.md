# Agent 0.5 Geo Reference Refresh Policy

## Purpose

Define when Agent 0.5 should:

1. continue normally with partial ZIP mapping
2. warn about unmapped ZIPs
3. trigger a geo reference dataset refresh workflow

This keeps `geo_zip_reference` as a shared national lookup while allowing new offers to be processed safely.

## Core Rule

New offer or new ZIP list does **not** automatically require a new `geo_zip_reference` dataset.

Agent 0.5 should:

1. reuse the existing `geo_zip_reference`
2. detect unmapped ZIPs
3. escalate only when unmapped coverage exceeds a threshold

## Runtime Behavior

### Normal Path

For every new offer or updated ZIP list:

1. normalize and store the ZIP list in `offer_geo_coverage`
2. look up ZIPs in `geo_zip_reference`
3. continue scoring using mapped ZIPs
4. write `missing_zip_count` and `missing_zip_ratio` into candidate reasoning

### Escalation Thresholds

Trigger a geo-reference refresh workflow when either threshold is met:

- `missing_zip_count >= 25`
- `missing_zip_ratio > 0.05`

These are the default thresholds.

## Refresh Modes

### Script Mode

If the geo reference importer exists:

- package script: `npm run import:geo-zips`

Agent 0.5 may invoke it automatically when the thresholds are met and auto-refresh is enabled.

### Prompt Mode

If the importer script is not available, Agent 0.5 should generate a ready-to-run implementation prompt and surface it in logs/reasoning.

This allows the system to escalate even before the importer is implemented.

## Operational Notes

- Refreshing `geo_zip_reference` is an infrastructure action, not an offer-specific cache action.
- `offer_geo_coverage` and `deployment_candidates` remain per-offer caches.
- The refresh path should be rare and threshold-driven.

## Success Criteria

This policy is successful if:

- Agent 0.5 keeps processing offers even with some unmapped ZIPs
- high unmapped coverage is detected deterministically
- a refresh workflow is available without blocking all offer processing
